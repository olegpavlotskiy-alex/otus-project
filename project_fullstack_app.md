# Full‑Stack приложение, сгенерированное AI с нуля

**Описание:**
Создать full‑stack приложение (backend + frontend + БД), используя AI для генерации архитектуры, кода и инфраструктуры. 
Проект стартует с описания функционала, а не с кода.

**Результаты:**
- Работающее приложение с минимум 4‑5 ключевыми фичами
- Тесты, документация и CI pipeline
- Демо‑развёртывание (Docker-compose, локально)

**Технологии:**
- Python
- PostgreSQL
- Docker, GitHub Actions

Далее будут представлены несколько вариантов приложений.


## Общие требования для всех вариантов

**Функциональные (базовые):**
- CRUD основных сущностей с валидацией на клиенте и сервере
- Поиск и фильтрация по ключевым полям
- Дашборд с визуализацией данных (графики / статистика)
- Пагинация списков
- Адаптивная вёрстка (desktop + mobile)

**Нефункциональные:**
- Seed‑данные: приложение запускается с реалистичным набором демо‑данных
- REST API с OpenAPI/Swagger документацией (в виде живой веб-страницы, а не доки в проекте)
- Не менее 10 unit/integration тестов
- CI pipeline (GitHub Actions): lint + тесты; сборка Docker‑образа — опционально
- Всё приложение запускается одной командой `docker compose up` (backend + frontend + БД)
- файл README.md инструкцией по запуску (формат - markdown)
- Файл ARCHITECTURE.md с описанием архитектуры и плана разработки (начните с него, формат - markdown)
- (Опционально) Ссылка на видео-демонстрацию работы приложения может упростить и ускорить проверку
- Файл REPORT.md с вашим описанием истории создания проекта, с какими ключевыми проблемами вы столкнулись, как решали. Ваши удачные и неудачные шаги. Добавляйте туда данные по мере работы, не в самом конце. Формат - markdown.
- Проект создан в новом, пустом GIT репозитории. Существует адекватная история коммитов (не всё разом).

### Проект: Трекер личных финансов

Общие требования плюс:

**Функциональные:**
- Транзакции: сумма, дата, категория, описание, тип (доход/расход)
- Справочник категорий (иконки, цвета) с возможностью CRUD
- Месячные бюджеты: установка лимита на категорию, прогресс‑бар расхода
- Дашборд: круговая диаграмма расходов по категориям, линейный график динамики за 6 месяцев, топ‑5 категорий
- Фильтрация транзакций по дате, категории, сумме (от/до)
- Импорт транзакций из CSV‑файла (маппинг колонок при загрузке)
- Экспорт отфильтрованных транзакций в CSV
- Повторяющиеся транзакции (ежемесячные подписки, зарплата) — автосоздание по расписанию
- Мультивалютность: выбор основной валюты, хранение курса на момент транзакции
- Регистрация / авторизация (JWT), два предзаполненных аккаунта (например, личный и семейный); каждый пользователь видит только свои транзакции и бюджеты
- Лог изменений: каждое создание, редактирование и удаление транзакции или бюджета фиксируется с указанием пользователя, времени и значений до/после; журнал доступен для просмотра

**Seed‑данные:** 200+ транзакций за 6 месяцев, 12 категорий, 3 бюджета, 2 пользователя

## Критерии оценки

- Выполнены все **общие требования** (CRUD, поиск, дашборд, пагинация, адаптивная вёрстка, тесты, CI, Docker Compose, OpenAPI, README)
- Выполнены все **функциональные требования выбранного варианта**
- Приложение поставляется с **seed‑данными** указанного объёма
- **Docker Compose** — приложение запускается в чистом окружении одной командой `docker compose up` без дополнительной настройки (блокирующий критерий: не выполнено — отправка на доработку)
- **AI‑процесс разработки** — ARCHITECTURE.md создан до начала кодирования (проверяется по git‑истории); REPORT.md содержит описание использованных AI‑инструментов и минимум один показательный пример промпта с кратким разбором

## Процесс сдачи

1. **Репозиторий** — студент предоставляет ссылку на GitHub‑репозиторий
2. **Запуск** — преподаватель выполняет:
   ```bash
   git clone <repo> && cd <repo>
   docker compose up
   ```
3. **Проверка функционала** — преподаватель открывает приложение в браузере и проходит по основным сценариям варианта (seed‑данные уже загружены)
4. **API** — преподаватель открывает Swagger UI по адресу `http://localhost:<port>/docs` (или `/redoc`, `/api-docs`) и проверяет эндпоинты
5. **CI** — преподаватель проверяет, что pipeline в GitHub Actions проходит (lint + тесты + сборка)
6. **Демо (5 мин)** — студент показывает приложение, объясняет архитектуру и ключевые решения

## Советы
1. Самый ая частая причина отправки на доработку - что то пошло не так на этапе docker compose up. Рекомендуется поверить запуск в "чистом" окружении пержде чем сдавать.
